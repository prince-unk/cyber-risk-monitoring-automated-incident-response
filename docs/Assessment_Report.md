# SSH Brute Force Detection & Automated Response Pipeline

## Assessment Report

---

## Project Overview

This project demonstrates a simple Security Operations (SOC) workflow that detects SSH brute-force attacks using centralized log collection and Python automation.

The goal was to simulate how a security analyst can collect authentication logs from a Linux server, identify suspicious login attempts, evaluate their severity, and automatically generate reports that can help during an incident investigation.

The entire pipeline was built in a small virtual lab consisting of a monitoring server and a target server.

---

## Objectives

The objectives of this project were to:

- Configure centralized log collection using rsyslog.
- Simulate an SSH brute-force attack with Hydra.
- Detect suspicious authentication attempts from collected logs.
- Calculate a risk level based on the number of failed login attempts.
- Automatically generate incident reports and executive summaries.
- Build a complete detection-to-reporting workflow using Python.

## Project Workflow

The project follows the workflow below:

1. Hydra performs an SSH brute-force attack against the target server.
2. The target server records authentication failures.
3. rsyslog forwards authentication logs to the monitoring server.
4. The detection engine extracts failed SSH login attempts.
5. The risk engine assigns a risk score based on the number of failed attempts.
6. The response engine automatically generates incident documentation.
7. The main pipeline executes all stages in sequence.

---

## Detection Logic

The detection engine searches authentication logs for failed SSH login attempts.

For every detected event, it extracts:

- Timestamp
- Source IP Address
- Username
- Number of failed attempts
- Event type

The extracted information is stored inside a JSON file that acts as the input for the remaining stages of the pipeline.

---

## Risk Assessment Logic

Risk is determined using the number of failed authentication attempts.

| Failed Attempts | Risk Level | Risk Score |
| --------------- | ---------- | ---------- |
| 1–3             | Low        | 25         |
| 4–7             | Medium     | 60         |
| More than 7     | High       | 95         |

This approach helps prioritize incidents before generating reports.

## Key Findings

During testing, the monitoring server successfully detected SSH authentication failures generated by Hydra.

The pipeline correctly:

- Collected centralized authentication logs.
- Detected failed SSH login attempts.
- Classified incidents according to risk level.
- Generated structured JSON output.
- Produced incident reports automatically.
- Created an executive summary for management review.

The simulated brute-force attack generated a High Risk alert due to the large number of failed authentication attempts.

---

## Challenges Faced

Some challenges encountered during development included:

- Configuring rsyslog to forward logs correctly.
- Parsing JSON data consistently between different scripts.
- Handling file formatting while passing data between pipeline stages.
- Ensuring every script produced compatible output for the next stage.

These issues were resolved through testing and validation of each stage individually before integrating them into the final pipeline.

---

## Conclusion

This project demonstrates a complete detection and reporting workflow for SSH brute-force attacks in a controlled lab environment.

Rather than stopping at attack detection, the project continues through risk assessment and automated report generation, showing how security events can be transformed into information that is useful for both technical teams and management.

Although this is a lab implementation, the overall workflow reflects the basic stages commonly found in Security Operations Centers (SOCs) and security monitoring environments.
